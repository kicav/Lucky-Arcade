# Production deployment

The provided production stack assumes `lucky-arcade-app` has already been generated by `setup-linux.sh` or an upgrade script.

## Architecture

```text
Internet / TLS proxy
        |
      Nginx
        |
    PHP-FPM app
      /     \
PostgreSQL  Redis
              |
        queue workers
              |
          scheduler
```

## Prepare environment

```bash
cp .env.production.example .env.production
cd lucky-arcade-app
php artisan key:generate --show
```

Copy the generated key into `APP_KEY` in `.env.production`, then replace every example password and URL.

## Start the stack

From the repository root:

```bash
docker compose \
  --env-file .env.production \
  -f docker-compose.production.yml \
  up -d --build
```

The example publishes Nginx on port 8080. Put a TLS-terminating reverse proxy or managed load balancer in front of it before public use.

## Validate

```bash
docker compose --env-file .env.production -f docker-compose.production.yml exec app php artisan arcade:doctor --strict
docker compose --env-file .env.production -f docker-compose.production.yml exec app php artisan wallets:reconcile
docker compose --env-file .env.production -f docker-compose.production.yml ps
```

## Deployment sequence

1. Back up the database.
2. Build and tag the new image.
3. Run migrations once.
4. Restart application and queue workers.
5. Run `arcade:doctor --strict`.
6. Run a smoke test against `/up`, login and one virtual-credit play.
7. Confirm queue and scheduler activity in Admin → System health.

## Required long-running processes

- PHP-FPM application.
- Queue worker.
- Scheduler invocation every minute.
- Nginx or another web server.

The Docker Compose file defines these as separate services so a queue restart does not interrupt web requests.
